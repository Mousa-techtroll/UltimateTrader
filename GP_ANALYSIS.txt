Assessment
  The strategy ideas are coherent for XAUUSD H1, but the implementation is not yet a clean “3-engine EA”. In practice it is a legacy multi-plugin EA with 3 engines appended to it, and several important controls are only partially wired or
  not wired at all.

  - The engines are registered after the legacy entry plugins, and UltimateTrader/Include/Core/CSignalOrchestrator.mqh:356 returns the first valid signal it finds. So the 3 engines are not the top-level decision-makers; they are lower-
    priority plugins behind engulfing, BB mean reversion, range box, breakout, crash breakout, file entry, and the older phase-3.4 plugins. See the registration order in UltimateTrader/UltimateTrader.mq5:294.
  - Engine-native qualityScore and engine_confluence are mostly telemetry, not true decision drivers. The orchestrator overwrites setupQuality, qualityScore, and riskPercent using a second text-based evaluator layer in UltimateTrader/
    Include/Core/CSignalOrchestrator.mqh:500.
  - Confirmation is on by default for all non-mean-reversion signals, so most engine trades are delayed one H1 bar. That especially matters for the session engine, whose edge is time-local.

  Engines

  - LiquidityEngine in UltimateTrader/Include/EntryPlugins/CLiquidityEngine.mqh is a priority cascade: Displacement -> OB Retest -> FVG Mitigation -> SFP. This is the best structured engine conceptually. Displacement is the strongest mode
    in the code: it requires an H1 sweep, a displacement candle, and SMC confluence, and it is the only liquidity mode allowed on volatile days. OB Retest and FVG Mitigation are weaker implementations than their names suggest: both rely
    more on context booleans and candle rejection than on precise zone tracking, so they behave more like “SMC-filtered rejection entries” than true execution-at-level models. SFP is the richest mode operationally, with fractal validation,
    volume gating, optional RSI divergence, and strong forensic logging, but it is disabled by default.
  - Liquidity-engine weaknesses: ScoreLiquidityLevel() uses GetSMCConfluenceScore(SIGNAL_LONG) for both directions in UltimateTrader/Include/EntryPlugins/CLiquidityEngine.mqh:1417, and the regime-based liquidity adjustment happens after the
    its constructor UltimateTrader/Include/EntryPlugins/CLiquidityEngine.mqh:64, while the EA-wide stop input is InpMinSLPoints = 800 in UltimateTrader/UltimateTrader_Inputs.mqh:66.
  - SessionEngine in UltimateTrader/Include/EntryPlugins/CSessionEngine.mqh tracks the Asian range, then trades London breakout, NY continuation, Silver Bullet, and London close reversal. Strategically this is coherent for gold. Silver
    Bullet is the cleanest mode because it really scans M15 FVG structure. London breakout and NY continuation are simpler momentum/continuation models using H1 closes and ATR buffers, not true retest models.
  - Session-engine weaknesses: it assumes server time is GMT unless a hidden gmt_offset is supplied; GetGMTHour() subtracts m_gmt_offset, but the constructor default is 0 and the EA never exposes broker offset calibration, so a non-GMT
    CSessionEngine.mqh:808. Also, despite receiving SetDayType() from the router each bar in UltimateTrader/UltimateTrader.mq5:798, the session engine does not use day type to gate entries; it only stores it into emitted signals.
  - ExpansionEngine in UltimateTrader/Include/EntryPlugins/CExpansionEngine.mqh is the cleanest engine architecture. Panic Momentum is a short-only bear-regime expansion trade, Institutional Candle Breakout is a proper state machine, and
    Compression Breakout is a BB-inside-Keltner squeeze release. This engine is the most internally consistent of the three.
  - Input hygiene is the biggest structural problem. Several inputs are dead or partly dead: InpTrailStrategy, InpEnableTrailOptimizer, InpTrailBETrigger, InpTrailBEOffset, and the older Group 8 trailing/TP controls are defined in
    UltimateTrader/UltimateTrader_Inputs.mqh:74 and UltimateTrader/UltimateTrader_Inputs.mqh:121, but the runtime always initializes and registers all six trailing plugins in UltimateTrader/UltimateTrader.mq5:397. InpModeKillMinTrades and
    InpModeKillPFThreshold are defined in UltimateTrader/UltimateTrader_Inputs.mqh:332, but I found no SetModeKillParams() calls, so changing them does not change engine behavior. InpEnableDynamicWeights and the strategy-weight inputs are
    also not wired. InpUseStructureTargets is ignored because CAdaptiveTPManager defaults use_structure_targets = true in its constructor UltimateTrader/Include/Core/CAdaptiveTPManager.mqh:79 and the EA never calls Configure() after
    inside the risk plugin; signal.riskPercent acts more like a cap than a true base-risk input. There is a second issue: volatility risk is logged after lot sizing in UltimateTrader/Include/Core/CTradeOrchestrator.mqh:256, but unlike the
    200 EMA adjustment it does not trigger a lot-size recalculation.
  - The advertised TP/trailing workflow is not what the live code does. ManageOpenPositions() in UltimateTrader/Include/Core/CPositionCoordinator.mqh:955 implements TP0 partial close and early invalidation, then trailing and exit plugins.
    There is no TP1 or TP2 partial-close state machine in live management. More importantly, execution sends only one broker TP, tp1, through UltimateTrader/Include/Core/CTradeOrchestrator.mqh:288 into UltimateTrader/Include/Execution/
    CEnhancedTradeExecutor.mqh:1326 and then Buy()/Sell() in UltimateTrader/Include/Execution/CEnhancedTradeExecutor.mqh:1501. So in normal operation the broker closes the remaining position fully at TP1; TP2 is mostly an internal field,
    not a real staged target.
  - The trailing system is therefore misrepresented. The EA prints one “active” trailing strategy, but UltimateTrader/Include/Core/CPositionCoordinator.mqh:1227 loops over every registered trailing plugin and applies every “better” stop
    proposal. Since all plugins are enabled by default through CTradeStrategy, the real behavior is “whichever plugin suggests the tightest improving SL wins next”, not “use InpTrailStrategy”. This makes optimization of a single trailing
    mode misleading.
  - Plugin-level auto-kill and dynamic weighting are present on paper but effectively dead. RecordPluginTradeResult() exists in UltimateTrader/Include/Core/CSignalOrchestrator.mqh:168, but I found no call sites anywhere in the project, so
    plugin PF/drawdown never updates. Engine-mode auto-kill is live, but plugin-level auto-kill and rolling dynamic weights are not. Day routing has a similar issue: DAY_DATA exists, but UltimateTrader/Include/Core/CDayTypeRouter.mqh:39
    never emits it, so the liquidity engine’s “no trading on data day” branch is dead code.

  Bottom line: the 3 engines are strategically interesting and gold-oriented, but the implementation is still inconsistent end-to-end. The strongest live engine today is LiquidityEngine via Displacement; the cleanest architecture is
  ExpansionEngine; the highest implementation risk is SessionEngine because of time-base assumptions, missing confluence initialization, and day-router non-enforcement. The trailing and multi-target system is the weakest part of the EA: it
  is marketed as staged/selector-driven, but the live code is effectively TP0 + single broker TP + all trailing plugins competing.
  
  
  
  Issues found: 
  
  Critical

  - Rebuild the trailing and target workflow first. Empirically, the 25 TRAILING exits made +42.68R, while every non-trailing exit combined made -22.47R. The CSV also shows 0 TP_HIT exits, 4 exits exactly at OriginalTP1 that were still
    labeled TRAILING, 35 BREAKEVEN exits that are basically TP0 rescues, 73 BE_Before_TP1 cases, and 139/168 trades with modified stops. In code, all six trailing plugins are registered in UltimateTrader/UltimateTrader.mq5:397, all
    strategies default enabled in UltimateTrader/Include/PluginSystem/CTradeStrategy.mqh:26, the coordinator applies every improving stop in UltimateTrader/Include/Core/CPositionCoordinator.mqh:1227, and execution only sends one broker TP
    in UltimateTrader/Include/Core/CTradeOrchestrator.mqh:288 and UltimateTrader/Include/Execution/CEnhancedTradeExecutor.mqh:1326. I also found no production code setting tp1_closed=true or tp2_closed=true; only tests do. Fix: one selected
    trailing mode only, real TP0/TP1/TP2 state machine, real partial closes, and wired BE trigger/offset inputs.
  - Make the EA actually engine-first, or separate legacy and engine runs. Right now the “3-engine EA” claim is not what this backtest executed. Legacy plugins are registered before engines in UltimateTrader/UltimateTrader.mq5:294, and the
    orchestrator returns the first valid plugin signal in UltimateTrader/Include/Core/CSignalOrchestrator.mqh:356. That is why 156/168 trades came from legacy engulfing plus one liquidity mode. Fix: explicit priority/arbitration at engine-
    family level, or separate operation modes for legacy, engines, and hybrid.
  - Remove universal confirmation from time-sensitive strategies. The CSV shows ConfirmationUsed=YES on all 168 trades. In code, almost every non-mean-reversion signal is stored pending in UltimateTrader/Include/Core/
    CSignalOrchestrator.mqh:528, and confirmed positions are marked true in UltimateTrader/UltimateTrader.mq5:832. That is especially damaging for session breakouts: London was the worst session overall at -1.60R, London Breakout went 0/3,
    and NY Continuation went 0/2. Fix: mode-specific confirmation policy; skip or relax confirmation for session breakouts, NY continuation, and expansion releases.

  High

    Include/EntryPlugins/CLiquidityEngine.mqh:1431, and regime adjustment happens after the gate in displacement and SFP in UltimateTrader/Include/EntryPlugins/CLiquidityEngine.mqh:520 and UltimateTrader/Include/EntryPlugins/
    CLiquidityEngine.mqh:1180. Fix: direction-aware scoring and regime adjustment before thresholding.
  - Fix risk sizing so it uses the actual signal metadata. 85% of trades were labeled SETUP_A_PLUS, yet A+ expectancy was only about +0.125R, which means the quality layer is not discriminating well. Part of the reason is structural: the
    reduced risk interface fabricates SETUP_B_PLUS and PATTERN_NONE in UltimateTrader/Include/RiskPlugins/CQualityTierRiskStrategy.mqh:412, and volatility risk is adjusted after lot sizing in UltimateTrader/Include/Core/
    CTradeOrchestrator.mqh:256. Fix: use the signal-aware sizing path and recalculate lots after every risk adjustment.
  - Stop optimizing dead or misleading inputs. InpTrailStrategy, InpEnableTrailOptimizer, InpTrailBETrigger, InpTrailBEOffset, InpTP1Volume, InpTP2Volume, and InpBreakevenOffset are defined in UltimateTrader/UltimateTrader_Inputs.mqh:74 and

    SetModeKillParams() has no call sites. InpEnableDynamicWeights and InpWeightRecalcInterval are only defined, not used. Fix: wire them end-to-end or remove them from the exposed parameter surface.

  Medium

  - Normalize stop-distance handling across engines. EA-wide InpMinSLPoints = 800 is defined in UltimateTrader/UltimateTrader_Inputs.mqh:69, but liquidity defaults to 100 in UltimateTrader/Include/EntryPlugins/CLiquidityEngine.mqh:64,
    expansion defaults to 100 in UltimateTrader/Include/EntryPlugins/CExpansionEngine.mqh:72, and session hardcodes 100 in UltimateTrader/Include/EntryPlugins/CSessionEngine.mqh:121. Fix: pass the same min-stop input into every engine.
