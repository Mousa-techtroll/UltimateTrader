METHODOLOGY
-----------
Three independent analyses (CD, GP, GM) were evaluated against the actual
codebase (~80 files, ~15k lines) and the 168-trade backtest (Mar 2025 - Mar
2026). Each analyst brought different strengths:

  CD — Deep code + data forensics. Quantified every bug's R-impact using
       the trade CSV. Identified early invalidation as the #1 P&L destroyer.
       Strongest on: backtest statistics, per-session/day/hour breakdowns,
       MFE analysis, trailing system diagnosis.

  GP — Deepest code-level tracing. Found call-site gaps (auto-kill wiring,
       strategy recording), metadata fabrication in risk sizing, vol
       adjustment applied after lot calculation without recalculation.
       Strongest on: execution flow tracing, dead code identification,
       telemetry integrity, disciplined fix sequencing.

  GM — MQL5 platform expertise. Identified struct padding/CRC brittleness,
       ArrayResize performance, data sync risks. Proposed "Shadow Trail"
       concept. Strongest on: platform-specific risks, production
       robustness, conceptual strategy improvements.

──────────────────────────────────────────────────────────
 PHASE 1: EXIT/TRAILING SYSTEM (Dominant P&L driver)
──────────────────────────────────────────────────────────

The backtest proves: 25 TRAILING exits produce +42.68R (the entire alpha).
Everything else is breakeven or loss. Fixing the exit system has the highest
R-impact ceiling.

1.1  DISABLE OR FIX EARLY INVALIDATION
     Source: CD (primary, -26.90R), GM (conceptual critique)
     Evidence: 39 trades, 2.6% WR, -$1,158.60. The "loss avoided" metric
               is -26.90R — exits are WORSE than holding to SL.
     Files: CPositionCoordinator.mqh:1062-1108, UltimateTrader_Inputs.mqh
     Action (immediate): Set InpEnableEarlyInvalidation = false
     Action (better): Replace MAE/MFE symmetric check with structural
       invalidation per GM's suggestion:
       - Only exit if price CLOSES below the originating order block (for
         OB Retest / FVG trades) or below the Asian range (for session trades)
       - Remove the arbitrary 3-bar time window
       - Keep the MFE_R < 0.05 condition (truly dead trades) but raise
         MAE_R threshold to 0.80 (almost full-SL)
     Complexity: 1 input change (immediate) or ~40 lines (structural fix)
     Expected impact: +15 to +27R (CD estimate, verified against data)
     Confidence: VERY HIGH — directly measurable from existing backtest

1.2  WIRE InpTrailStrategy TO SELECT A SINGLE TRAILING PLUGIN
     Source: CD (primary), GP (verified), GM (noted)
     Evidence: InpTrailStrategy is dead. All 6 run. ATR at 1.35x overrides
               Chandelier at 3.0x. System captures only 16% of MFE.
               124.95R left on table across 168 trades.
     Files: UltimateTrader.mq5:400-425
     Action: After creating all trailing plugins, disable all except selected:
       for(int t = 0; t < g_trailingPluginCount; t++)
       {
          g_trailingPlugins[t].SetEnabled(false); // disable all
       }
       // Enable only the selected one:
       switch(InpTrailStrategy)
       {
          case TRAIL_ATR:        g_atrTrailing.SetEnabled(true); break;
          case TRAIL_SWING:      g_swingTrailing.SetEnabled(true); break;
          case TRAIL_PARABOLIC:  g_sarTrailing.SetEnabled(true); break;
          case TRAIL_CHANDELIER: g_chandelierTrailing.SetEnabled(true); break;
          case TRAIL_STEPPED:    g_steppedTrailing.SetEnabled(true); break;
          case TRAIL_HYBRID:     g_hybridTrailing.SetEnabled(true); break;
          case TRAIL_NONE:       break; // all disabled
       }
     Complexity: ~15 lines
     Expected impact: +15 to +30R (CD estimate). With Chandelier-only, more
       trades survive initial pullbacks to become runners. 35 BREAKEVEN exits
       with avg MFE of 0.946R could convert to TRAILING exits.
     Confidence: HIGH — the trailing exits are the proven alpha source

1.3  IMPLEMENT TP1/TP2 PARTIAL CLOSES IN POSITION COORDINATOR
     Source: CD (primary), GP (verified)
     Evidence: TP0 fires on 85 trades (50.6%), provides 64.8% of dollar
               profit. But broker closes 100% at TP1. TP2 never reached.
               InpTP1Volume and InpTP2Volume are dead inputs.
               52 trades (31%) reached 1.0R — potential TP1 zone.
     Files: CPositionCoordinator.mqh (ManageOpenPositions), Structs.mqh
     Action: Add TP1 and TP2 partial close logic mirroring the TP0
       implementation. Use R-multiple triggers:
       - TP1 at InpTP1Distance (1.3R): close 50% of REMAINING lots
       - TP2 at InpTP2Distance (1.8R): close 50% of REMAINING lots
       - Runner: remaining ~18.75% of original, managed by trailing
       - Remove the broker-level TP. Let trailing manage the final exit.
       - Fix the 115% specification: use remaining-lots not original-lots
       State machine transitions:
         INITIAL → TP0_HIT (at 0.5R) → TP1_HIT (at 1.3R) → TP2_HIT (at
         1.8R) → TRAILING (managed by selected trailing plugin)
     Complexity: ~80-100 lines (GP: "position state-machine extension")
     Expected impact: +5 to +12R. Converts 10-15 breakeven/SL trades into
       partial-profit + runner trades. The 35 breakeven exits that averaged
       0.946R MFE would capture partial at TP0 + more at TP1 if reached.
     Confidence: MEDIUM-HIGH — needs backtest verification

1.4  WIRE DEAD TRAILING/BE INPUTS
     Source: CD, GP (both verified)
     Files: CPositionCoordinator.mqh, trailing plugin files
     Action: Wire InpTrailBETrigger (0.8R) as explicit breakeven trigger
       threshold. Wire InpTrailBEOffset as the offset above entry for BE.
       Wire InpBreakevenOffset. These are currently dead.
     Complexity: ~20 lines
     Expected impact: Better control over breakeven behavior. Currently BE
       triggers whenever ANY trailing plugin's SL crosses entry — with a
       single trailing plugin (fix 1.2), this becomes more predictable.

1.5  FIX HYBRID TRAILING BREAKEVEN CAP
     Source: CD (verified at CHybridTrailing.mqh:255-256)
     Files: CHybridTrailing.mqh
     Action: Remove the cap that prevents SL from moving above entry.
       The cap at entry - 10 points makes Hybrid the most conservative
       strategy, defeating its purpose. Allow profit-locking above entry.
     Complexity: ~5 lines (delete the cap logic)

2.1  IMPLEMENT SIGNAL COLLECT-AND-RANK
     Source: CD (primary), GP ("signal arbitration")
     Evidence: 92.3% of trades from 2 patterns. Session Engine only 8 trades.
               Registration order determines priority, not quality.
     Files: CSignalOrchestrator.mqh (CheckForNewSignals)
     Action: Replace the sequential first-valid-wins loop with:
       1. Collect all valid signals from all plugins into a temp array
       2. Score each by setupQuality * qualityScore * engine_confluence
       3. Return the highest-scoring signal
     Complexity: ~30 lines
     Expected impact: +2 to +5R. Better signals win when multiple fire
       simultaneously. Engine signals (Silver Bullet at 83-85 quality) would
       beat Engulfing (82 quality) when both are valid.
     Confidence: MEDIUM — interaction effects unknown until backtested

2.2  ADD PER-PLUGIN CONFIRMATION POLICY
     Source: GP (unique insight: all 168 trades had ConfirmationUsed=YES)
     Evidence: Every trade including time-sensitive session breakouts and
               expansion releases waited 1 extra H1 bar for confirmation.
               This worsens R:R on time-sensitive setups.
     Files: CSignalOrchestrator.mqh, CEntryStrategy.mqh
     Action: Add a virtual method RequiresConfirmation() to CEntryStrategy.
       Override in each engine:
       - Legacy patterns (Engulfing, etc.): return true
       - CLiquidityEngine: return true (structural setups benefit)
       - CSessionEngine: return false (time-gated, delay is costly)
       - CExpansionEngine: return false (volatility is time-decaying)
       In the orchestrator, check plugin.RequiresConfirmation() instead of
       the global InpEnableConfirmation flag.
     Complexity: ~20 lines
     Expected impact: Session and Expansion engines get faster entry,
       potentially improving their signal count and R:R.
     Confidence: MEDIUM — those engines had too few trades to measure

2.3  REMOVE DOUBLE VOLATILITY ADJUSTMENT
     Source: CD (primary), GP (identified different facet)
     Evidence: GetVolatilityRiskMultiplier() applied in CQualityTierRiskStrategy
               Step 3 AND again in CTradeOrchestrator:259. The 4 VOLATILE
               trades had the BEST performance (PF 10.82, +0.93R avg) yet
               were the most under-sized by this bug.
     Files: CTradeOrchestrator.mqh:257-267
     Action: Delete the volatility adjustment block in CTradeOrchestrator.
             The risk strategy already handles it.
     ALSO (GP finding): The vol adjustment in CTradeOrchestrator applies
       AFTER lot calculation without recalculating lots. Even as a single
       application, the implementation is wrong — it should either:
       (a) Be removed (preferred — risk strategy handles it), or
       (b) Trigger lot recalculation if kept
     Complexity: Delete 8 lines
     Expected impact: +1 to +3R. Correct sizing on volatile-regime trades.

2.4  FIX RISK INTERFACE METADATA FABRICATION
     Source: GP (unique finding, verified)
     Files: CQualityTierRiskStrategy.mqh:412
     Action: The risk strategy fabricates SETUP_B_PLUS and PATTERN_NONE when
             it doesn't receive proper signal metadata. Pass the actual
             setupQuality and patternType from the EntrySignal through to
             the risk calculation.
     Complexity: ~15 lines
     Expected impact: Correct risk sizing based on actual signal quality
       rather than fabricated defaults.

2.5  ENABLE SHORT TRADING
     Source: CD (primary, verified: zero shorts in 12 months)
     Evidence: 100% long bias. Multiple blocking layers compound to prevent
               any short signal from passing validation AND sizing.
     Files: CSignalValidator.mqh, CQualityTierRiskStrategy.mqh,

     Action (in order of priority):
       a) Audit short risk multiplier: ensure it applies ONCE, not twice.

          CQualityTierRiskStrategy:106. Remove from orchestrator.

          the multiplicative chain from producing sub-minimum lots.
     Complexity: Medium (~40 lines across multiple files)
     Expected impact: +2 to +6R from capturing bearish moves.
     Confidence: MEDIUM — needs dedicated backtest

     Complexity: ~40 lines in CPositionCoordinator
