Priority 0: Make the Measurements Trustworthy

Fixes

- Record strategy and mode results from real exit paths so the CSV reflects actual contributors.

Expected impact
- No direct PnL boost.
- Very high decision quality impact because it turns later backtests into something trustworthy.

- CD supports the broader maintainability concern but is not the main source here.

Priority 1: Rewrite Exit Semantics Before Retuning Entries
Rationale
The backtest proved that the EA currently depends heavily on trailing outcomes while its staged-target language and input surface do not match live behavior.

Fixes
- Add an explicit trade lifecycle state machine: `INITIAL -> TP0 -> TP1 -> RUNNER -> FINAL`.
- Stop using a single broker TP as a stand-in for a staged exit system.
- Implement real partial closes for TP0 and TP1 where configured.
- Persist target-hit state on the position so management logic is not reconstructing intent indirectly.
- Separate the concepts of protective TP0 rescue, true breakeven, TP1 realization, and runner management in both execution and reporting.

Expected impact
- Highest structural expectancy impact.
- Reduces the current mismatch between what the EA says it is doing and what it actually does.
- Makes later trailing optimization meaningful instead of misleading.

Source blend
- GP correctly identified this as the central defect.
- GM adds the useful idea that early invalidation should be more structural, which fits naturally once the state machine exists.
- CD’s general warnings about architecture debt support this area, but GP remains the main authority.

Priority 2: Make Trailing a Selected Policy, Not a Competition Between Plugins
Rationale
The current behavior effectively allows multiple trailing plugins to compete by repeatedly tightening stops, which makes `InpTrailStrategy` misleading and optimization results difficult to interpret.

Fixes
- Register only the active trailing plugin selected by input or by an explicit mode-to-trailing map.
- Persist the chosen trailing mode per position at entry time.
- Wire break-even trigger and offset parameters into the actual trailing behavior instead of leaving them as mostly cosmetic inputs.
- Distinguish between TP-driven stop movement, breakeven logic, and runner-trailing logic.
- Add logging that states which trailing policy actually owned the trade.

Expected impact
- Very high on consistency and interpretability.
- Likely high on live robustness because stop movement becomes deterministic and auditable.

Source blend
- GP identified the live trailing defect precisely.
- GM’s defensive perspective strengthens the case for deterministic ownership.
- CD adds little unique value here.

Priority 3: Replace First-Hit Signal Arbitration With Candidate Ranking
Rationale
A 3-engine EA cannot be evaluated fairly if the runtime still behaves like an older plugin stack that accepts the first valid signal in registration order.

Fixes
- Collect all valid candidates for the bar instead of returning immediately on the first passing plugin.
- Rank candidates using explicit policy: engine family, regime fit, session fit, setup quality, and conflict rules.
- Add explicit runtime modes such as `legacy`, `engines_only`, and `hybrid_ranked` so the system’s operating model is not ambiguous.
- Make engine family priority a deliberate configuration rather than an accident of registration order.
- Log why the winning candidate beat the rejected ones.

Expected impact
- High trade-mix impact.
- Required if the user wants a real 3-engine EA instead of a mostly legacy/FVG system.
- Improves the validity of later engine-specific tuning.

Source blend
- GP got this exactly right.
- GM partly weakens itself by assuming the engines are cleaner than they are, but its general desire for more coherent strategy routing is still useful.
- CD’s maintainability criticism indirectly supports simplifying the arbitration model.

Priority 4: Split Confirmation Policy by Strategy Type
Rationale
Universal confirmation is too blunt. It is especially damaging for time-sensitive breakouts and session-local moves.

Fixes
- Keep confirmation only where it is demonstrably beneficial, such as slower mean-reversion or selected structural setups.
- Allow session breakouts, expansion releases, and continuation trades to use no confirmation or a lighter confirmation rule.
- Record confirmation policy in the trade log for later expectancy analysis.
- Backtest with confirmation segmented by mode instead of globally enabled.

Expected impact
- High for trade timing and fill quality.
- Likely strongest on session and expansion strategies.

Source blend
- GP is strongest here.
- GM is compatible with this once its broader structural assumptions are corrected.
- CD does not materially improve this recommendation.

Priority 5: Rehabilitate the Session Engine Before Trusting It
Rationale
Session logic is one of the most fragile areas because it is sensitive to exact time alignment, confluence initialization, and day-type context.

Fixes
- Expose broker/server-to-GMT normalization clearly, or auto-detect it safely and log the result.
- Initialize engine confluence correctly instead of letting it behave like a near-zero placeholder.
- Enforce day-type gating instead of merely passing day labels through signals.
- Pass the real global minimum-stop policy into the engine instead of relying on engine-local defaults.
- Add immediate mode-level kill switches for London Breakout and NY Continuation until the fixed version proves itself.
- Revalidate Silver Bullet separately from breakout modes because it is structurally different.

Expected impact
- Moderate immediate PnL containment.
- High reduction in false confidence around session logic.

Source blend
- GP accurately identified the live defects.
- GM materially improves this section with stronger GMT handling and operational hardening.
- CD contributes only at the broad level of “revalidate weak branches.”

Priority 6: Harden the Liquidity Engine, Especially FVG Mitigation
Rationale
Liquidity/FVG logic carried much of the tested profitability, which makes code correctness here disproportionately important.

Fixes
- Correct directional confluence scoring so short and long paths do not share a long-only confluence lookup.
- Apply regime adjustments before entry threshold checks rather than after a trade is already eligible.
- Make FVG mitigation more context-aware: session filter, regime filter, and possibly a narrower volatility or displacement quality requirement.
- Improve zone bookkeeping so order-block and gap states age out or invalidate more explicitly.
- Tighten the relationship between sweep, displacement, and mitigation so the engine is not just taking generic rejection candles with SMC labels.

Expected impact
- Moderate to high because this is the dominant live mode family.
- Higher quality of signal selection, lower accidental overtrading.

Source blend
- GP correctly surfaced the verified scoring defects.
- GM adds useful stricter filtering and context hardening.
- CD contributes the reminder that structural objects should have better lifecycle management, but its exact scoring claims should not be trusted without revalidation.

Priority 7: Clean Up the Risk and Quality Pipeline
Rationale
The current quality labels and risk sizing path overstate precision. The risk plugin loses too much real signal metadata and can make low-information defaults look more intelligent than they are.

Fixes

- Align setup-evaluator string matching with the actual comments emitted by engine strategies.
- Stop inflating too many trades into the top quality bucket if expectancy does not justify it.
- Either make pattern-confidence truly pattern-aware or remove the misleading abstraction.

Expected impact
- Moderate.

Source blend

Priority 9: Add Data Integrity and Synchronization Guards
Rationale
Even a logically correct strategy becomes brittle if it trades on stale or incomplete data.

- Guard array indexing assumptions near boundaries and early history.
- Skip trading decisively when required data is incomplete instead of silently proceeding.
- Add logging that differentiates a skipped trade due to missing data from a filtered trade due to market logic.

Expected impact
- Low direct PnL in clean backtests.
- High robustness in live operation and in less perfect data environments.

- Use mode-specific invalidation rules instead of a one-size-fits-all rule.
